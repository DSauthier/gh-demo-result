Thank you for the detailed context and goals. Here’s a summary and a recommended strategy for your demo workflow, especially to make phase 3 repeatable and robust:

Demo Workflow Summary
Phase 1:

Build a simple shopping cart app with a deliberate bug (invoice value can be altered by the user).
Push to GitHub and deploy to an EC2 instance using GitHub Actions.
Phase 2:

Create a GitHub issue describing the bug, as a QA team would.
Phase 3:

Use Copilot Chat to:
Read and understand the issue.
Fix the bug in a development branch (e.g., dev-branch-1).
Test and deploy the fix to a test/dev environment.
Merge and deploy to production if successful.
Be able to repeat this phase easily for demo purposes.
Recommended Strategy for Repeatable Demos
Branching Model:

Use at least three branches:
main (production)
dev-branch-1 (for fixes and testing)
buggy-demo (always contains the buggy version for resets)
Environments:

Set up two environments in your deployment workflow:
Production: Deploys from main
Dev/Test: Deploys from dev-branch-1
Resetting for Repeated Demos:

To reset, simply:
Reset main and dev-branch-1 to the buggy state from buggy-demo.
Re-deploy to both environments.
Optionally, automate this reset with a script or GitHub Action.
Demo Flow:

Start with the buggy app in both environments.
Create the issue.
Use Copilot Chat to fix the bug in dev-branch-1.
Test in the dev environment.
